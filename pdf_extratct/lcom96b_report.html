📐 Abstração & Instabilidade

Fórmulas de Robert C. Martin

A = Na / Nc   |   I = Ce / (Ca + Ce)   |   D = |A + I − 1|

A = Abstração (tipos abstratos / total)  |  I = Instabilidade (deps saída / total deps)  |  D = Distância da Sequência Principal
Na = interfaces + classes abstratas  |  Ca = acoplamento aferente  |  Ce = acoplamento eferente

📝 Análise de Conascência do Projeto

1. Conascência de Nome (CoN) — Nível: Aceitável
Com 31 referências cross-module contra 146 intra-module, o projeto mantém a maioria das dependências dentro dos módulos (82.5%), o que é positivo. O módulo loginpage é o mais referenciado (hub central), com 5 módulos dependendo dele. Isso é esperado por ser o módulo de domínio principal.

2. Conascência de Tipo (CoT) — Nível: Atenção
Das 172 dependências totais, 90.7% são a tipos concretos e apenas 9.3% a abstrações. Isso viola o Princípio da Inversão de Dependência (DIP). Recomendação: introduzir interfaces nos módulos security (A=0.0) e common (A=0.0), que não possuem nenhuma abstração.

3. Conascência de Significado (CoM) — Nível: Bom
Apenas 2 referências cross-module a Enums. Os 4 enums do projeto (AuthProvider, TransactionType, JobStatus, NamePackageEnum) estão bem encapsulados nos respectivos módulos.

4. Conascência de Posição (CoP) — Nível: Moderado
11 classes possuem construtores com mais de 3 parâmetros. Destaque para JobResponse (11 params) e TransactionResponse (8 params). Embora records/DTOs aceitam mais parâmetros por natureza, classes como PdfController (7 deps injetadas) e AuthService (5 deps) podem indicar responsabilidades excessivas. Considere usar o padrão Builder ou dividir as classes.

Resumo Geral: O projeto apresenta acoplamento controlado entre módulos, com a maioria das dependências concentradas intra-módulo. Os principais pontos de melhoria são: (1) aumentar a taxa de abstração (DIP) especialmente nos módulos security e common, e (2) reduzir a conascência de posição nas classes com muitos parâmetros no construtor.
